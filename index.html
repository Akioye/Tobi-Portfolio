🛒 Self-Healing E-Commerce Platform
Pods die. Kubernetes brings them back. Under 30 seconds. Every time.
✓ Completed
Problem
How do you build a microservices platform that recovers from failures automatically, with no human intervention, and prove it works under real conditions?
Solution
Three FastAPI microservices deployed to AWS EKS via Helm, with 2 replicas per service across 2 availability zones. Kubernetes reconciliation loop detects pod failure and schedules replacement instantly. Proved with chaos engineering and live Grafana video.
Architecture
GitHub Actions CI/CD → ECR → EKS (t3.small, 2 nodes, 2 AZs) → Helm charts → kube-prometheus-stack → Grafana. Terraform provisioned 20 AWS resources including VPC, subnets, NAT Gateway, EKS cluster, ECR repos, and bastion host.

02 / 03
🤖 GitOps Zero-Touch Deployment Pipeline
Push code. Everything else is automated. ArgoCD does the rest.
✓ Completed
Problem
Manual deployments are slow, error-prone, and leave no audit trail. How do you build a pipeline where Git is the single source of truth and no human ever touches the cluster directly?
Solution
Two-repo GitOps pattern: infra-gitops holds app code and CI/CD pipeline, k8s-manifests holds Kubernetes manifests watched by ArgoCD. Push code → GitHub Actions builds and tags image with commit SHA → updates staging manifest → ArgoCD syncs cluster. Staging auto-deploys. Production is manually promoted.
Architecture
GitHub → GitHub Actions → ECR → k8s-manifests → ArgoCD → EKS (staging + production namespaces). Sealed Secrets encrypts sensitive data for safe Git storage. Terraform manages all AWS infrastructure. Slack notifications on every deployment success or failure.
Tech Stack
AWS: EKS, ECR, VPC, NAT Gateway
GitOps: ArgoCD, Sealed Secrets
CI/CD: GitHub Actions
IaC: Terraform
App: Node.js
Notifications: Slack Webhooks

Lessons Learned
Two-repo separation is not just convention — it is a real security boundary. ArgoCD credentials never need access to source code. selfHeal: true means Git always wins; manual kubectl changes are reverted within 3 minutes. Kubernetes version upgrades must be sequential — you cannot skip minor versions. Next additions: Helm charts for cleaner multi-environment config, ArgoCD Image Updater, Argo Rollouts for canary deployments.

🏗️ Reusable Terraform Module Library
Security and best practices baked in by default. Not optional.
✓ Completed
Problem
Copy-pasting infrastructure code across environments leads to drift, security gaps, and inconsistency. How do you make secure defaults automatic across dev, staging, and production?
Solution
Five reusable Terraform modules — VPC, IAM, S3, RDS, EKS — with security baked in. The S3 module always blocks public access. VPC always enables flow logs. IAM always uses least privilege. Three fully isolated environments with non-overlapping CIDRs. GitHub Actions runs parallel terraform plan on all three on every PR.
Architecture
Bootstrap (S3 + DynamoDB) → 5 modules → dev (10.0.0.0/16) / staging (10.1.0.0/16) / prod (10.2.0.0/16). Remote state per environment. DynamoDB locking prevents concurrent applies. CI matrix: Plan dev + Plan staging + Plan prod in parallel. All plans passed = required merge gate.
Tech Stack
IaC: Terraform ~1700 lines
State: S3 + DynamoDB locking
Security: IRSA, OIDC, KMS encryption
Database: RDS Postgres 15.4
CI/CD: GitHub Actions matrix
AWS: EKS, VPC, RDS, S3, IAM

Key Design Decisions
Separate directories vs Terraform workspaces
Decision: Directories. Workspaces share backend — a mistyped workspace select before apply could hit prod. Directories make the target explicit.
CI plans only — never applies
Decision: A compromised token or merge mistake should never auto-destroy production. Humans review plans. Humans apply.
single_nat_gateway = true in dev/staging
Decision: Saves ~$32/month per environment. Dev is disposable. Production uses one NAT per AZ for HA.
deletion_protection + skip_final_snapshot in prod
Decision: Prod RDS cannot be deleted without deliberate override. A terraform destroy cannot silently wipe production data.
Lessons Learned
A module library is only valuable if the defaults are secure — otherwise you just have shared bad practices. IRSA changes the blast radius of a compromised pod from "entire EC2 node permissions" to "that pod's specific permissions only." DynamoDB locking per state key means dev and staging can apply simultaneously without blocking each other. Next additions: module versioning with git tags, Secrets Manager integration, multi-AZ RDS in prod, Cluster Autoscaler.
